**고객의 개인정보는 은행 수준의 암호화로 보호되며, 본인 이외에는 누구도 열람할 수 없는 구조입니다.**

> 이 건물은 국제 보안 인증(ISO 27001)을 받은 시설입니다.

## 2. 누가 데이터를 볼 수 있나요? (접근 통제)

### Row Level Security (RLS) — "내 서랍은 나만 열 수 있다"

모든 데이터 테이블에 **행 수준 보안(RLS)**이 적용되어 있습니다.

| 사용자 유형 | 볼 수 있는 데이터 |
|------------|------------------|
| **일반 사용자** | 본인의 프로필, 신청서, 결제내역, 상담 내역만 |
| **기관 담당자** | 소속 기관의 학생/직원 상담 내역만 |
| **관리자 (변호사)** | 담당 고객의 상담 내역 |
| **최고 관리자** | 전체 데이터 (법적 의무 이행 목적) |

### 쉬운 비유

> 아파트 우편함과 같습니다.
> 각 세대(사용자)는 자기 우편함만 열 수 있고,
> 관리사무소(관리자)만 전체 우편함에 접근할 수 있습니다.
> **옆집 우편함을 여는 것은 기술적으로 불가능합니다.**

## 3. 로그인은 어떻게 보호되나요? (인증)

| 방식 | 설명 |
|------|------|
| **Google 소셜 로그인** | Google의 보안 인프라를 그대로 활용 (2단계 인증 포함) |
| **이메일/비밀번호** | 비밀번호는 **bcrypt** 방식으로 암호화 저장 (원본 복원 불가) |
| **세션 관리** | 로그인 토큰(JWT)은 자동 만료, 로그아웃 시 즉시 삭제 |

### 쉬운 비유

> 비밀번호는 "일방통행 암호화"로 저장됩니다.
> 원래 비밀번호 → 암호화된 문자열 (되돌릴 수 없음)
> 해커가 데이터베이스를 탈취해도 원본 비밀번호는 알 수 없습니다.

## 4. 문서(파일)는 어떻게 보호되나요?

### 이중 보안: 암호화 + 접근 제한

```
[사용자 기기] → AES-256 암호화 → [서버 저장]
                 ↑                      ↑
           사용자만 가진 열쇠        암호화된 상태로 보관
```

| 보안 계층 | 설명 |
|-----------|------|
| **전송 암호화** | 모든 통신은 HTTPS(TLS 1.2+)로 암호화 |
| **파일 암호화** | AES-256-GCM 방식 (군사/은행 수준 암호화) |
| **열쇠 관리** | 암호화 키는 사용자 기기에만 존재, 서버에 전송되지 않음 |
| **접근 URL** | 서명된 URL로만 접근 가능, 일정 시간 후 자동 만료 |

### 쉬운 비유

> 여권 사본을 택배로 보낸다고 생각해보세요.
> 1) 문서를 자물쇠가 달린 금고에 넣고 (AES-256 암호화)
> 2) 금고를 방탄 호송차로 배달합니다 (HTTPS 전송)
> 3) 열쇠는 본인만 가지고 있습니다 (클라이언트 키 관리)
> **택배 기사(서버)도 금고 안의 내용을 볼 수 없습니다.**

## 5. 결제 정보는 안전한가요?

| 보안 항목 | 적용 여부 | 설명 |
|-----------|-----------|------|
| PCI DSS 준수 | O | Toss Payments가 카드 정보를 직접 처리 |
| 중복 결제 방지 | O | 고유 거래키(Idempotency Key)로 이중 결제 원천 차단 |
| 금액 검증 | O | 서버에서 결제 금액 재검증 |
| 감사 로그 | O | 모든 결제 시도를 5년간 기록 보관 |

### 쉬운 비유

> 카드번호는 저희 서버를 **거치지 않습니다**.
> Toss Payments(PG사)가 직접 카드 정보를 처리하고,
> 저희는 "결제 완료 확인서"만 받습니다.
> 편의점에서 카드를 긁을 때 편의점이 카드번호를 저장하지 않는 것과 같습니다.

## 10. 자주 묻는 질문 (FAQ)

### Q. Supabase가 미국 회사인데, 데이터가 미국에 저장되는 건 아닌가요?

> Supabase는 AWS 인프라를 사용하며, 리전 선택이 가능합니다.
> 전송 중 데이터는 항상 암호화(TLS)되며, 저장 시에도 AWS 관리형 암호화가 적용됩니다.
> 또한 개인정보 처리 위탁 계약(DPA)을 통해 한국 법률 수준의 보호를 보장합니다.

### Q. 해커가 데이터베이스에 침입하면 개인정보가 유출되나요?

> **아닙니다.** 세 가지 방어선이 있습니다:
> 1) 비밀번호는 bcrypt로 단방향 암호화 — 원본 복원 불가
> 2) 문서 파일은 AES-256으로 암호화 — 열쇠 없이 해독 불가
> 3) RLS 정책으로 인증 없는 데이터 조회 자체가 차단

### Q. 직원이 퇴사하면 고객 데이터에 접근할 수 있나요?

> 관리자 권한은 `admins` 테이블에서 관리되며,
> 퇴사 시 해당 계정을 비활성화하면 즉시 모든 접근이 차단됩니다.
> RLS 정책이 매 요청마다 권한을 확인하므로 우회가 불가능합니다.

### Q. 카드 정보가 서버에 저장되나요?

> **절대 저장되지 않습니다.**
> 카드 정보는 PCI DSS 인증을 받은 Toss Payments가 직접 처리하며,
> 저희 서버에는 "결제 성공 여부"와 "거래 ID"만 기록됩니다.

### Q. GDPR(유럽 개인정보보호법) 요건도 충족하나요?

> 현재 서비스는 한국 개인정보보호법을 기준으로 운영되며,
> 데이터 처리 동의, 열람/삭제 권리, 데이터 이동권 등
> GDPR의 핵심 요건과 동등한 수준의 보호를 제공합니다.
